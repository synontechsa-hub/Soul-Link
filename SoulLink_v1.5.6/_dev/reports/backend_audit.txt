1. Executive Summary

This report presents a full system audit of the SoulLink backend (Legion Engine v1.5.6), covering 64 Python source files spanning the API layer, data models, business logic, middleware, services, and test suite. The audit was conducted across five domains: naming consistency & init file hygiene, WebSocket & auth security, backend stability, bug catalog, and a prioritized implementation plan.



The codebase shows clear architectural ambition and a structured refactoring effort in progress ("Normandy SR-2"), but the migration is incomplete. Legacy models co-exist with new ones without deprecation guards; hardcoded secrets and UUIDs are scattered across the production code; and several critical security gaps exist in the WebSocket and SSV reward pipelines.

2.2 app/models/__init__.py — Imports Deleted/Non-Existent Models

Severity

Detail

CRITICAL

models/__init__.py imports UserSoulState and SoulRelationship. These models are DEPRECATED and have been superseded by LinkState, but they still exist as table=True models. This means SQLModel will attempt to register two competing schemas for the same or overlapping tables at startup, causing SQLAlchemy MetaData conflicts or duplicate table registration errors.



Imports that will crash or cause conflicts:

3. Security Audit — WebSocket, JWT, OAuth, SSV



3.1 WebSocket Security

Severity

Issue

CRITICAL

Token still accepted via query param (?token=...). The comment says 'moved to handshake', but the code in websocket.py still reads token = websocket.query_params.get('token') and uses it if present. Tokens in query params are logged in server access logs, Nginx logs, and CDN logs — a severe credential exposure vector.

HIGH

No per-IP connection rate limiting on WebSocket endpoint. The WebSocketManager enforces MAX_CONNECTIONS_PER_USER = 5, but an attacker can open 5 connections from each of N IPs (or spoofed IPs), flooding the server. The HTTP rate limiter (slowapi) does not apply to WebSocket upgrade requests.

HIGH

websocket_manager.disconnect() is not awaited in the finally block of websocket_endpoint. The disconnect() method is synchronous (def disconnect) and removes the connection from the dict. However, if the underlying WebSocket close raises, the metadata dict and connections dict can desync, leaving zombie entries.

MEDIUM

After websocket.accept() is called (before auth), any rejection via websocket.close() still shows the server accepted the TCP handshake. This makes the endpoint exploitable for port-scanning and connection-exhaustion even without valid auth. Auth should be validated before accept().

MEDIUM

WebSocket message size is unbounded. A malicious client can send a multi-megabyte JSON blob which json.loads() will attempt to parse in the async loop. No max_size guard exists.

LOW

The /ws/stats endpoint uses Depends(require_architect_role) but require_architect_role is an HTTP dependency — its credentials come from HTTPBearer security header. For an HTTP GET endpoint wrapping a WebSocket stats view, this is fine, but it means the endpoint is HTTP-only and the architect dependency is being re-validated fresh rather than using the cached WS session.



3.2 JWT / Supabase Auth Security

Severity

Issue

CRITICAL

ARCHITECT_UUID hardcoded as a string literal in 7 separate places: dependencies.py, chat.py, souls.py (×2), sync.py, and others. This is a security anti-pattern — a developer accidentally committing to a public repo or rotating the UUID will cause inconsistent behavior across files. The settings.py has architect_uuid as an Optional field but it is NEVER used — the hardcoded literal takes precedence.

HIGH

Token cache TTL is 60 seconds, but Supabase JWT expiry is typically 3600s. If a token is revoked/invalidated in Supabase (logout, password reset), the cached auth will continue to work for up to 60 seconds. This window is short but non-zero. More importantly, the cache key is only 32 hex chars of SHA256 — while collision probability is low, it should use the full hash.

HIGH

require_architect_role() has a logical bug: if _check_architect_role_sync returns the ARCHITECT_UUID, we skip the HTTPException — but we still call cache_service.set(cache_key, user_id) where user_id could be None if _check_architect_role_sync returned None but user_id == ARCHITECT_UUID. The condition 'if user_id == ARCHITECT_UUID' doesn't short-circuit the None case correctly — it sets None in cache if both conditions fail.

MEDIUM

JIT user provisioning in get_current_user() does not validate that the Supabase UUID matches a UUID v4 format before inserting into the database. A malformed or injected user_id could create a user record with an unexpected primary key format.

MEDIUM

The supabase client singleton is created at module import time. If settings.supabase_url or settings.supabase_anon_key are empty strings (misconfigured .env), create_client() may succeed but produce a non-functional client — future auth calls will return None without raising, silently failing open.



3.3 Ad Server-Side Verification (SSV) Security

Severity

Issue

CRITICAL

hmac.new() does not exist — it should be hmac.new() in Python's hmac module... actually hmac.new() is the correct API but the variable name shadows the module: 'import hmac' then 'expected_signature = hmac.new(...)'. This is actually correct Python — but the HMAC signature logic is broken: AppLovin SSV callbacks come as GET requests with query parameters (not POST JSON body), yet the code expects a POST payload. The /ads/verify-reward endpoint should be a webhook receiver matching AppLovin's actual SSV spec.

HIGH

When applovin_ssv_secret == 'mock_secret' (the default), signature validation is entirely skipped (is_valid = True). This means in production if the env var is not set, ANY caller can grant themselves unlimited stability boosts by hitting /ads/verify-reward with any user_id they know.

HIGH

The /ads/verify-reward endpoint requires user auth (get_current_user), which is correct for the identity check. However, AppLovin SSV callbacks come from AppLovin's servers, not from the user's device. AppLovin cannot provide a user Bearer token — so this endpoint as designed cannot actually receive real SSV callbacks.

MEDIUM

No replay attack prevention on SSV callbacks. The same (user_id, timestamp, signature) triple could be submitted multiple times to grant multiple stability boosts. An idempotency check against AdImpression records should block duplicates.



3.4 CORS & General HTTP Security

Severity

Issue

HIGH

In main.py, if settings.environment == 'production' but settings.production_frontend_url is None, the code falls back to an empty list for allowed_origins — the comment says 'Fallback' but the actual CORS behavior with an empty allow list may default to blocking all origins OR the truncated code may have a different fallback. This is a production CORS misconfiguration risk.

MEDIUM

X-Forwarded-For header is trusted without validation in get_real_ip() for rate limiting. An attacker can spoof any IP by setting the X-Forwarded-For header, bypassing per-IP rate limits entirely. This should only be trusted if traffic goes through a known proxy.






4. Backend Stability Scan



4.1 Database & Session Management

Severity

Issue

CRITICAL

The sync Engine is created using settings.database_url which is a postgresql:// URL. SQLModel's create_engine with a sync driver on a Postgres URL will work, but pool_size=20 + max_overflow=10 is configured on the SYNC engine as well as the ASYNC engine — meaning 60 total connections are potentially held (30 per engine × 2), easily saturating Supabase's 60-connection default limit in the free tier.

HIGH

time_manager.py calls await self.session.commit() AFTER sending WebSocket messages (websocket_manager.send_to_user). If the commit fails, the WebSocket notification has already been sent, causing a client/server state split — clients believe time advanced but the database did not change.

HIGH

location_manager.py calls await self.session.commit() then broadcasts location_update via WebSocket. Same race condition: if WS broadcast fails, the DB is committed. If DB commit fails, WS was never sent (correct order here), but the move_to method catches the exception and returns an error tuple — yet the caller in map.py uses the result AFTER the commit: loc = await session.get(Location, location_id). If the session was rolled back, this get() call is in an invalid session state.

HIGH

The sync query events (@event.listens_for(Engine, 'before_cursor_execute')) are registered on the generic Engine class, not the specific engine instance. This means these listeners fire on ALL SQLAlchemy engines created in the process (including any test engines), potentially causing test pollution.

MEDIUM

get_async_session() does not set autocommit=False or autoflush=False explicitly. With expire_on_commit=False set, lazy loading on detached instances after commit will silently return stale data.



4.2 Concurrency & Async Issues

Severity

Issue

CRITICAL

brain.py calls self.client.chat.completions.create(...) — this is a SYNCHRONOUS call from Groq's SDK inside an async FastAPI handler. There is no await and no run_in_executor wrapper. This BLOCKS the entire uvicorn event loop for the duration of the LLM call (typically 1-10 seconds), making all concurrent requests queue behind it.

HIGH

narrator.py: generate_chronicle() is an async staticmethod but instantiates NarratorService() inside itself just to access srv.client (the Groq lazy property). The client property checks hasattr(self, '_client') — but since a new instance is created every call, the lazy caching never works. A new Groq() client is created on every chronicle generation.

HIGH

CacheService uses a class-level dict (_memory_cache) shared across all instances and threads. With asyncio + multiple workers (uvicorn --workers N), each worker process has its own memory cache — keys set in worker A are not visible in worker B. Users hitting different workers get inconsistent cached data. Redis must be used for multi-worker deployments.

MEDIUM

WebSocketManager._connections and _connection_metadata are plain Python dicts with no asyncio.Lock protection. Concurrent coroutines accessing connect() and disconnect() simultaneously can cause dict size changes during iteration (RuntimeError) or partial state corruption.



4.3 Error Handling Gaps

Severity

Issue

HIGH

location_resolver.py has the resolve_location() method defined TWICE with the same signature (duplicate @staticmethod def). Python will silently use the second definition, discarding the first. This means the Priority 1 (LinkState manual override) logic is DEAD CODE — it is defined in the first (overridden) method body.

HIGH

backup_service.py: _backup_table() is truncated in the snapshot — the model_dump() call is incomplete. If model_dump() fails for any record (e.g. non-serializable datetime), the entire backup silently fails with a logged error. No partial backup recovery.

MEDIUM

brain.py: in the except block for session.commit() failure, the code calls print(f'❌ TRANSACTION FAILED: {e}') instead of logger.error(). This print call will not appear in structured logs in production.

MEDIUM

chat.py: PersonaService.create_default_persona() is called with await session.flush() internally but no subsequent commit. The persona object is returned and used in brain.generate_response(), where persona.last_used = datetime.utcnow(); session.add(persona) is called — but if the persona was only flushed (not committed) and the session rolls back, the JIT-created persona is lost silently.

MEDIUM

health.py: readiness_check() references JSONResponse without importing it from fastapi.responses. The import is done inside the health_check() function (correct) but the readiness_check() function uses it in the if ws_count > 500 branch without importing it in scope — this will NameError when the overloaded condition triggers.



4.4 Cache & Memory Issues

Severity

Issue

HIGH

souls.py _get_soul_blueprint() caches SQLModel ORM objects (Soul, SoulPillar) in the cache dict. SQLModel objects are tied to a SQLAlchemy session. Once the session closes, accessing any lazy-loaded relationship on the cached object raises DetachedInstanceError. Only serialized dicts (model.model_dump()) should be cached.

MEDIUM

CacheService pruning removes the 100 OLDEST INSERTION ORDER keys, not the 100 oldest by expiry time. Keys near expiry should be pruned first — the current logic can remove recently-set valid keys while keeping stale entries.

MEDIUM

map.py caches the geography output (including mutable present_souls and time_slot fields) for 1 hour. However, soul locations change every time advance_time_slot is called. The code tries to patch this by re-injecting soul_locs into the cached_geo list in-place, but the cache is then NOT updated with the patched data — the next request will get the original stale present_souls.
